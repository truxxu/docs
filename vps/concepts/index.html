Linux Concepts

1. The Linux Mindset​

Linux treats almost everything as a file or a process. Configuration lives in text files, software is managed through package managers, and most administration is done from the command line.

On a VPS, this means you will usually:

  • Connect through SSH
  • Edit config files
  • Install packages
  • Start and stop services
  • Inspect logs
  • Manage users and permissions
  • Troubleshoot from the shell
tip

The goal is not to memorize hundreds of commands — understand the core building blocks and know which commands to reach for.

2. Connecting to Your VPS​

Most VPS management starts with SSH.

ssh root@your_server_ip

If you use a non-default port:

ssh -p 2222 root@your_server_ip

If you use an SSH key:

ssh -i ~/.ssh/my_key root@your_server_ip
warning

Avoid working as root for everyday tasks once initial setup is complete. Create a normal user and use sudo when elevated privileges are needed.

3. Knowing Where You Are in the Filesystem​

Linux has a single directory tree rooted at /.

DirectoryPurpose
/Root of the filesystem
/homeUser home directories
/rootRoot user's home directory
/etcConfiguration files
/varVariable data: logs, caches, web content, databases
/usrInstalled programs and libraries
/bin, /sbinEssential system binaries
/tmpTemporary files
/srvService data (on some systems)
/optOptional third-party software

Useful navigation commands:

pwd          # show current directory
ls           # list files
ls -la       # show hidden files and detailed info
cd /etc      # change to /etc
cd ~         # go to your home directory
cd ..        # go up one level

4. Creating, Moving, Copying, and Deleting Files​

touch newfile.txt
mkdir newfolder
mkdir -p app/newfolders        # create nested directories
cp -r myfolder myfolder_copy   # copy a directory
mv oldname.txt newname.txt     # rename
mv file.txt /tmp/              # move to another location
rm file.txt                    # delete a file
rm -r myfolder                 # delete a directory
danger

Be careful with deletion, especially as root.

rm -rf some_directory

This force-removes files recursively and can destroy a system if used carelessly.

5. Reading File Contents Quickly​

You will often inspect config files and logs.

cat file.txt             # print the whole file
less file.txt            # scroll through a file
head -n 20 file.txt      # show the first 20 lines
tail -n 20 file.txt      # show the last 20 lines
tail -f /var/log/syslog  # follow new lines in real time

6. Editing Files from the Terminal​

On a VPS, you will often edit text-based config files. Common editors:

  • nano — easier for beginners
  • vim — more powerful, steeper learning curve
nano /etc/ssh/sshd_config
tip

If you are new, nano is enough to get started.

7. Users, Groups, and sudo​

Linux is multi-user by design. On a VPS, this matters for security.

whoami                   # show current user
id                       # show user and group info
sudo command             # run a command with elevated privileges
adduser deploy           # create a new user
usermod -aG sudo deploy  # add user to the sudo group
passwd deploy            # change a user's password

:::tip Good practice Use a regular user with sudo rather than direct root login for daily administration. :::

8. Understanding File Permissions​

Every file and directory in Linux has an owner, a group, and a set of permissions.

ls -l

Example output:

-rw-r--r-- 1 user user 1234 Apr 21 10:00 file.txt
drwxr-xr-x 2 user user 4096 Apr 21 10:00 myfolder

How to Read the Permission String​

Take -rw-r--r-- and break it into parts:

CharactersMeaning
First charFile type (- = file, d = directory)
Next 3Owner permissions
Next 3Group permissions
Last 3Others permissions

So -rw-r--r-- means:

  • - → regular file
  • rw- → owner can read and write
  • r-- → group can only read
  • r-- → everyone else can only read

What r, w, and x Mean​

For files:

PermissionMeaning
rRead the file
wModify the file
xExecute the file as a program

For directories:

PermissionMeaning
rList the directory contents
wCreate, delete, or rename files inside it
xEnter the directory and access items inside
note

A directory usually needs x to be usable, not just r.

Why Permissions Use Numbers​

Permissions are often set numerically using chmod:

chmod 644 file.txt
chmod 755 script.sh

Each digit represents owner, group, and others. Each digit is built from:

ValuePermission
4read
2write
1execute

Add them together to get the desired permission.

Example — chmod 644 file.txt:

DigitCalculationResult
64 + 2rw-
44r--
44r--

Result: -rw-r--r--

Example — chmod 755 script.sh:

DigitCalculationResult
74 + 2 + 1rwx
54 + 1r-x
54 + 1r-x

Result: -rwxr-xr-x

Common Permission Examples​

644 — normal file

chmod 644 notes.txt   # -rw-r--r--

Use for regular text and config files. Owner can read and edit; everyone else can only read.

600 — private file

chmod 600 ~/.ssh/id_rsa   # -rw-------

Use for sensitive files like private SSH keys. Only the owner can read and write.

755 — executable file or accessible directory

chmod 755 deploy.sh         # -rwxr-xr-x
chmod 755 /var/www/myapp    # drwxr-xr-x

Use for scripts you want to run and for directories others need to access. Owner has full access; group and others can read and enter, but not modify.

700 — private executable or private directory

chmod 700 backup.sh      # -rwx------
chmod 700 ~/private-dir  # drwx------

Only the owner has access.

Why Directories Often Use 755​

Directories need x to be entered. A directory with 644 is usually wrong:

chmod 644 mydir   # drw-r--r-- ← can't be entered normally

Common directory permissions:

  • 755 for shared-access directories
  • 700 for private directories

Changing Permissions with chmod​

Numeric mode:

chmod 644 file.txt
chmod 755 script.sh

Symbolic mode:

chmod u+x script.sh    # add execute for owner
chmod g-w file.txt     # remove write for group
chmod o-r secret.txt   # remove read for others

Symbolic references: u = user/owner, g = group, o = others, a = all.

Changing Ownership with chown​

chown user:user file.txt
chown -R www-data:www-data /var/www/app

This matters a lot for web servers. If your app files are owned by the wrong user, Nginx, Apache, or your app process may not be able to read or write what it needs.

Practical Rules​

PermissionUse Case
644Most regular files
755Directories and executable scripts
600Secrets like private keys
700Private directories or scripts

:::danger Avoid 777

chmod 777 file.txt   # rwxrwxrwx

Everyone can read, write, and execute. On a VPS, that is a security problem, not a solution. Fix the actual owner or assign only the permissions that are needed. :::

9. Processes and System Load​

A process is a running program. When something is broken on a VPS, checking processes is one of the first steps.

ps aux         # list running processes
top            # show live CPU and memory usage
htop           # more readable version of top (if installed)
pgrep nginx    # find process IDs by name
kill 1234      # send termination signal to process 1234
kill -9 1234   # force-kill a process
note

Use kill -9 sparingly — it does not allow graceful shutdown.

10. Services and systemd​

Most modern Linux VPS distributions use systemd to manage services.

systemctl status nginx    # check whether a service is running
systemctl start nginx     # start
systemctl stop nginx      # stop
systemctl restart nginx   # fully restart
systemctl reload nginx    # reload config without full restart (if supported)
systemctl enable nginx    # start automatically at boot
systemctl disable nginx   # disable autostart

This is essential for web servers, databases, SSH, firewalls, and application processes.

11. Logs and Troubleshooting​

Logs are one of your best tools when something fails.

With systemd:

journalctl -u nginx         # all logs for nginx
journalctl -u nginx -n 50  # last 50 lines
journalctl -u nginx -f     # follow in real time

Common log files in /var/log:

/var/log/syslog
/var/log/auth.log
/var/log/nginx/access.log
/var/log/nginx/error.log

Typical troubleshooting flow:

  1. Check service status
  2. Read recent logs
  3. Inspect config files
  4. Test configuration
  5. Restart or reload the service

Example for Nginx:

nginx -t              # test config
systemctl reload nginx

12. Package Management​

Software on Linux is usually installed through a package manager. On Debian/Ubuntu:

apt update          # refresh package lists
apt upgrade         # install available updates
apt install nginx   # install software
apt remove nginx    # remove software
tip

Keeping packages updated is a basic security practice on a VPS.

13. Networking Basics​

ip a                       # show network interfaces and IP addresses
ip r                       # show routes
ss -tulpn                  # show listening ports and the processes using them
ping 8.8.8.8               # test connectivity
curl ifconfig.me           # show public IP from the server
curl http://localhost:3000 # test a local web service

If a website is down, check whether the process is running and whether the expected port is actually listening.

14. DNS and Hostname Basics​

A VPS may have both a hostname and one or more IP addresses.

hostname           # show hostname
hostnamectl        # detailed hostname info
nslookup example.com
dig example.com

These help verify your server's hostname, domain name resolution, and whether a domain points to the correct IP. DNS mistakes are common when deploying websites.

15. Disk Space and Memory​

Resource checks are routine on a VPS.

df -h              # show disk usage by filesystem
du -sh /var/www/app  # show directory size
free -h            # show memory and swap usage

If your server behaves strangely, low disk space or low memory is often the cause.

16. Searching and Filtering Output​

grep "error" /var/log/syslog     # search text in a file
grep -R "server_name" /etc/nginx # search recursively
find /var/www -name "*.log"      # search by filename

These are essential when dealing with configs, logs, and large directory trees.

17. Shell Redirection and Pipes​

Redirect output to a file:

echo "hello" > file.txt       # overwrite
echo "another line" >> file.txt  # append

Pipe output into another command:

cat /var/log/syslog | grep nginx
ss -tulpn | grep 80

The pipe | sends the output of one command into another. A common pattern: generate output, filter it, and inspect only what matters.

18. Environment Variables​

echo $HOME         # view one variable
env                # view all variables
export APP_ENV=production  # set one temporarily

This will only last for the current shell session unless added to shell config files or service definitions.

19. Exit Codes and Why They Matter​

Commands return an exit code:

  • 0 → success
  • non-zero → failure

Check the last command's exit code:

echo $?

This matters in scripts and automation. A command may print nothing but still fail, so exit codes are often more reliable than appearance alone.

20. Basic Scripting Mindset​

You do not need to be a shell scripting expert to manage a VPS, but simple scripts save time. Store them in .sh files.

#!/bin/bash
apt update && apt upgrade -y
systemctl restart nginx

Key ideas:

  • Scripts automate repeated tasks
  • && means run the next command only if the previous one succeeded
  • Always test scripts carefully before using them on production servers

21. Safe Habits When Working on a VPS​

tip

A few habits prevent a lot of damage:

  • Check your current directory before deleting files
  • Test config changes before restarting services
  • Use sudo deliberately, not casually
  • Keep backups of important config files
  • Inspect logs before guessing
  • Avoid changing many things at once
  • Update the system regularly
  • Prefer least privilege for users and services

Example — back up a config before editing:

cp /etc/nginx/nginx.conf /etc/nginx/nginx.conf.bak
